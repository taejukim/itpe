- 역할 및 사용 기술 등을 간략하게 기입
- 만약 잘 아는 내용이라면 Sub로 참여했더라고 어느정도 부풀려서 적는 것도 전략
- 절때 위축되면 안되고 나 잘났어요 를 뽐내는 문서임
- 장황하게 쓰지 않고 임팩트 있게 단어나 짧은 문장으로 서술
	- e.g. ) 선물하기 서비스 매주 정기배포 운영
		- - 메인 엔지니어로 Testcase 작성 및 리소스 할당 주관
		- - 선물하기 서비스 기본 당연품질에 기여


실력있는 QA의 조건
- QA Fundamental
  - QA가 필요한 이유
    - **고객이 장애를 만나지 않게 하는것, 불편함이 없어야 하는 것**
    - **고객의 재산과 생명을 침해하지 않는 것 < 이게 최우선**
  - 그것을 위해 어떤 일들을 해왔는지?
    - 테스트를 하지 않고도 제품의 품질을 확보 할 수 있다면 그게 최고
    - 자동화 테스트는 메인이 아님
  - 당연품질과 매력품질
    - 제품의 존재 자체인 당연품질
      - 타협할 수 없는 것들, 당연품질이 미흡하다면 매력품질은 존재할 수 없음
    - 마케팅 소구점, 신기능 등으로 설명할 수 있는 매력품질
- 꼼꼼함과 투명함
  - 촘촘한 테스트 케이스 작성
  - 업무에 대한 투명함
  - 꼼꼼한 문서 작성, 업무관리
- 미래지향적
  - 개인시간을 활용해서라도 미래에 대한 투자
  - 당장의 워라밸 보단 미래 성장을 위한 투자가 필요함
  - 현재에 만족하지 않고 끊임없이 앞으로 나가는 것이 필요
- 기술적인 것들
  - 현재 도메인에 대한 기술 외에 다양한 기술에 대한 호기심이 필요
  - 검증이 불가한 것을 검증할 수 있게도 만들 수 있어야 함
- 선택과 집중
  - 모든 영역에 대해 동일한 Effort로 검증하는 것은 바람직하지 않음
  - 취약한 부분, 중요한 부분 등은 조금더 강력하게 검증
  - 지금까지 결함이 없었던 영역, 중요하지 않은 부분, 사용이 적은 영역 등은 스킵하거나 가볍게 검증
  - 과거 이력, 경험적, 통계적 분석을 통해 의사결정 할 수 있음
QA 프로세스
  - 기획 단계 :  미연방지 활동, 기획서에 모순이나, 결함이 없는지
  - 구현 단계 : 개발자 테스트 결과 입수 
  - Q Gate : 모두가 합의된 기준이 필요 > 해당 기준에 부합하는지 하지 않는지 판단 > 다만 유연하게
  - QA :
    - 테스트 케이스 작성
    - 테스트 케이스 리뷰(내부, 외부)
    - 테스트 케이스 수행
    - 리그레션 테스트 수행
    - Sign off
  - 릴리즈 : 릴리즈 후 모니터링 (원하는 형상으로 배포된것인지?)
  - 회고 KPT (Keep, Problem, Try)

* 작성이 필요한 문서들
  1) QA Plan 구성요소
     1) QA 목적
     2) 일정(중요)
        1) 각 스텝별 일정 (시작 일정과 종료 일정을 명확하게)
        2) 각 스텝의 담당자, 기대되는 산출물을 기록
     3) 산출물
        1) 기획리뷰 회의록, 테스트 케이스 리뷰 회의록, 테스트 케이스 등
     4) 담당자
        1) 기획, 개발, QA 담당자 기록 (누가 이 프로젝트를 진행하는지)
     5) 테스트 대상
     6) 테스트 제외 대상
        1) 테스트 불가 영역, 테스트 Skip 영역 등 기록
     7) 테스트 환경, 도구, 빌드 등
        1) 제품의 버전, 환경, 검증 도구 등을 기록
     8) 기타..
  2) QA Report
     1) 산출물
        1) 테스트 케이스 수행결과
     2) 일정
        1) 실제로 진행된 일정을 기록
     3) 특이사항
        1) QA 진행 중 특이사항, 일정 지연, 의사결정 등의 내용을 기록
     4) Sign off
        1) 어떤 근거로 OK Sign을 한 것인지 기록
           2) e.g.) n개 케이스 수행 후 잔여이슈 없어 배포 가능
  3) Testcase
     1) 구성요소
        1) Testcase ID : 유니크한 식별자
        2) Testcase summary
        3) 분류 : 여러 방식이 있음 
           4) 분류 > 기능으로 2단계로 분류를 작성하고 Test procedure, Expect result 작성
           5) N Depth 작성 > Expected result
        6) Test procedure
        7) Test Expected result
        8) Priority : 우선순위
           9) 중요도, 비기능/기능 
           10) P1 ~ P5
              1) P1 : 기본 기능 - 명세에 있는 happy case
              2) P2 : 복합 기능 - 명세에 있는 특이사항
              3) P3 : 다양한 입력이 필요한 케이스
              4) P4 : 비기능 테스트
              5) P5 : 사용성
        11) Requirement ID
        12) Create User
        13) CreatedAt
        14) UpdatedAt
        15) Last modifier
        16) isAutomated
  4) 기획리뷰 회의록
     1) 프로젝트 배경
     2) 주요 변경사항
     3) Test 범위, 방법 확인
     4) 일정
        1) 개발일정
        2) QA 일정
        3) 배포/출시 일정
     5) 기타 특이사항
     6) 걱정점, QA가 중점적으로 봐주었으면 하는 영역, 미연방지 활동 등
  5) Testcase 리뷰 회의록
     1) 테스트 케이스 설계 공유
     2) 테스트 방법 논의
     3) 걱정점
6) * 자동화 테스트
  * 가장 중요한 것은 아래와 같은 것들
  * 자동화 테스트로 메뉴얼 테스트를 모두 대체할 수 없다.
  * 지속하지 못하는 자동화 테스트 프로젝트는 쓸모 없다.
  * 유지보수성이 최우선이다.
* 자동화 테스트 구축 프로세스
  - 인터페이스 확보
    - Web : Selenium, App : appium, Linux : ssh , serial, 남땜, 등등
  - 각 동작의 정의
    - Web/APP : API 또는 웹 접근, 다른 것들은 내부 프로토콜
  - 자동화 테스트 시나리오 작성
  - 사용 기술 선정 (Script 작성 전에만 하면됨)
  - Script 작성
  - 적용 
  - 효과 확인
- 필요한 기술
  - CI/CD : jenkins 등, 자동화테스트를 파이프라인을 통해 수행하기 위한 도구
  - Docker/Linux/shell script : CI/CD를 구축하기 위해 알고 있으면 좋은 것들
  - HTTP(S) : 거의 대부분 웹, 앱 서비스에서 사용되는 프로토콜
    - get, post, delete, put, session, cookie, SSL Handshake
    - RESTful API
      - uri : /v1/blog/
        - get - 게시글 조회
        - post - 게시글 등록
- Language : Python, Java, Typescript(testcafe)
- Framework : Pytest, Junit, Selenium, Appium, cypress
- Design pattern, Architecture : POM, Singleton, Builder, Factory 등
