## QA 사전과제 준비
- 품질 Fundamental 관련해 상황을 주고 해결 방법에 대한 열린 문제
	- 부족한 리소스, 모순되는 상황 등 → 합리적인 해결방안 제시 (선택과 집중, Data Driven 의사결정, 프로세스 간소화, 커뮤니케이션, Shift left 등)
- 주어진 요구사항을 보고 Testcase 설계
	- 아무 것도 없는 상황에서 Testcase의 주요 구성요소, 설계 기법 등 확인
	- Testcase 식별자, Requirement 식별자 등 추가
- 주어진 Markup이나, API Spec 등을 활용한 자동화 테스트 구현 → 보통 위 요구사항과 연결되는 경우가 많음
	- Pseudo code도 가능
	- Selenium, Playwright, Appium, Pytest, Junit5등 기본적인 것들 작성

### QA Fundamental
  - 고객이 장애를 만나지 않게 하는것, 불편함이 없어야 하는 것
  - 고객의 재산과 생명을 침해하지 않는 것 < 이게 최우선
  - 그것을 위해 어떤 일들을 해왔는지?
    - 테스트를 하지 않고도 제품의 품질을 확보 할 수 있다면 그게 최고
- 당연품질과 매력품질
  - 제품의 존재 자체인 당연품질
    - 타협할 수 없는 것들, 당연품질이 미흡하다면 매력품질은 존재할 수 없음
  - 마케팅 소구점, 신기능 등으로 설명할 수 있는 매력품질
* QA 프로세스 Setup
  - 기획 단계 : 미연방지 활동, 기획서에 모순이나, 결함이 없는지, 테스트 방법, 제약사항, 제외 항목
  - 구현 단계 : 개발자 테스트 결과 입수 -> Unit test 결과
  - Q Gate : 모두가 합의된 기준이 필요 > 해당 기준에 부합하는지 하지 않는지 판단 > 다만 유연하게
  - QA :
    - 테스트 케이스 작성
      - Test case 구성 요소
	      - ID 유니크 식별자
	      - Requirement 식별자
	      - Summary : 테스트 요약
	      - 분류 : 기능 분류
	        - e.g. ) 선물하기 > 선물함 > 주문내역 > 주문내역 상세
	      - Procedure : 테스트 시나리오
	      - Expected result : 기대결과
	      - 판정 (Pass/fail)
	      - 우선순위(H, M, L/P1 ~ P5)
    - 테스트 케이스 리뷰(내부, 외부)
    - 테스트 케이스 수행 (1 ~ 5day)
    - 리그레션 테스트 수행
    - Sign off
  - 릴리즈 : 릴리즈 후 모니터링 (원하는 형상으로 배포된것인지?)
  - 회고 KPT (Keep, Problem, Try) -> Action item 도출 -> 다음 Project 적용

### 자동화 테스트
* 자동화 프로세스
  - 사용 기술 선정
  - 인터페이스 확보
  - 각 동작의 정의
  - 자동화 테스트 시나리오 작성
  - Script 작성
  - 적용 
  - 효과 확인
- 필요한 기술적인 역량
  - CI/CD
    - Jenkins Pipeline을 구축하고 원하는 순서 대로 환경 구축 및 테스트 실행 > 리포팅
    - Shared library를 활용해 Pipeline도 형상관리하고 보다 효과적으로 적용
  - Docker
    - 환경이슈를 완전히 해결
    - 복잡한 환경구축을 쉽게 해결 할 수 있는 솔루션
    - Docker-compose 등으로 좀더 쉽게 여러 컨테이너 구성 가능
  - Linux
    - CI/CD, Docker 등을 활용하기 위해서는 Linux 관련된 지식이 필요
    - 파일 핸들링, 패키지 설치, 환경변수 설정, shell script 등 
    - 터미널과 친해지기
  - HTTP(필수)
    - 모든 앱, 웹 서비스의 기초가되는 프로토콜
    - GET, POST, PUT, DELETE 역할을 확실하게 알아야 함
    - Cookie, Session, Status code (e.g. 200 OK, 201, 301, 400, 404, 500, 503)
  - Language
    - python, java, typescript
  - Framework
    - playwright, pytest, selenium, appium/appium-mcp, junit
- 자동화 테스트 프로젝트 아키텍쳐
  - Sample project : [Qwazii-qa/dab_project](https://github.com/Qwazii-qa/dab_project)
  - 관련 영상 : https://youtu.be/Ez_9Gcy_BuM
  - Test와 인터페이스(src)를 완전하게 분리
    - Test class에는 인터페이스와 관련된 것들이 있으면 안됨
    - Test는 Testbase를 상속받아 전체 Test class에 공통으로 적용될 로직을 작성
      - e.g. 인터페이스 초기화, After, Before class, 공통 Assertion 로직등
    - src에는 인터페이스를 연동을 위한 것들을 추상화해 환경, 인터페이스 종류에 따라 공용으로 사용
      - 인터페이스 객체와 동작(API) 클래스를 분리하고 각각의 영역에만 집중할 수 있도록 구성
      - properties를 사용해 원하는 테스트 데이터를 수집하고 project 객체의 속성으로 전달
      - 프로젝트 전역에서 project 객체를 사용할 수 있도록 Testbase에서 로드
  - Page Object Model pattern, Screen Play pattern, API wrapper, Builder pattern, Factory pattern, Singleton pattern 등 디자인 패턴을 활용해 유지보수성을 극대화
  - 이후 BDD(cucumber-java, behave-python) 도구를 통해 자연어로 테스트 시나리오를 작성 > 수행  TestDrivenDev
    - given, when, then, gherkin syntax
