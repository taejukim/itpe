## 작성 원칙

### Ground Rules
1. Reference 기준 내용으로만 작성 (PDF 읽기 불가로 웹검색 활용)
2. 각 토픽별 독립된 .md 파일 생성
3. 통합 설명 가능 시 통합 생성
4. 모든 토픽에 태그 추가:
   - #정보관리기술사 (필수)
   - 각 도메인 Tag (e.g., #SW공학, #아키텍처, #테스트)
   - 카테고라이징 가능한 분류 태그
   - **태그는 한 줄로 통합 작성** (예: `#정보관리기술사 #SW공학 #아키텍처`)
5. 핵심만 요약, 줄글 말고 불렛 포인트로 작성
6. Text 기반 구성도 작성
7. **섹션 간 빈 줄 제거**: 컴팩트한 형식 유지 (`## 정의\n- 내용\n## 키워드`)
8. **섹션 순서 준수**: 정의 → 키워드 → 암기법 → 특징 → 목적 → 구성요소 → 구성도 → 연관 토픽
9. **회독 체크리스트 작성 금지**: 파일 하단에 체크리스트 섹션 추가하지 않음
10. **특정 헤더 형식 사용 금지**: `======================` 형식의 헤더 사용하지 않음

### 특징 작성 원칙

# 서브노트 파일 구조 변경 작업

## 작업 개요
모든 서브노트 파일(`.md`)의 구조를 일관되고 간결한 형식으로 변경하는 작업을 수행했습니다.

## 주요 변경 사항

### 1. 회독 체크리스트 삭제
- **삭제 대상**: 각 파일 하단에 있던 `## 회독 체크리스트` 섹션 및 체크박스 항목들
- **예시**:
  ```markdown
  ## 회독 체크리스트
  - [x] 1회
  - [ ] 2회
  - [ ] 3회
  - [ ] 4회
  - [ ] 5회
  ```
- **작업 범위**: 모든 서브노트 파일 (186개)

### 2. 키워드 및 암기법 섹션 위치 변경
- **변경 전**: 파일 하단에 위치
- **변경 후**: `## 정의` 섹션 바로 아래로 이동
- **새로운 구조**:
  ```markdown
  ## 정의
  ...
  
  ## 키워드
  ...
  
  ## 암기법
  ...
  
  ## 구성요소
  ...
  ```

### 3. 섹션 간 빈 줄 제거
- **목적**: 파일을 더 컴팩트하게 만들기
- **변경 전**:
  ```markdown
  ## 정의
  
  - 내용
  
  ## 키워드
  
  - 키워드
  ```
- **변경 후**:
  ```markdown
  ## 정의
  - 내용
  ## 키워드
  - 키워드
  ```

### 4. 태그 통합
- **변경 전**: 여러 줄에 걸쳐 태그가 분산
  ```markdown
  #태그1 #태그2
  #태그3
  ```
- **변경 후**: 모든 태그를 한 줄로 통합
  ```markdown
  #태그1 #태그2 #태그3
  ```

### 5. 특정 헤더 형식 삭제
- **삭제 대상**: 파일 상단에 있던 특정 형식의 헤더
- **예시**:
  ```markdown
  =======================
  Datamining(Data Mining)
  ```
- **작업 범위**: 해당 형식이 있는 일부 파일들
