## 정의
요구사항의 공학적 접근, 요구공학
- 사용자의 의도와 시스템이 제공해야 할 기능을 정확히 파악하여 문서화하고 관리하는 프로세스
- 프로젝트 실패의 주요 원인인 요구사항 변경과 불명확성을 해결하여 품질을 확보함

## 목적
- 사용자 및 이해관계자의 요구사항을 체계적으로 도출하고 명확히 하여 프로젝트의 범위 이탈 및 재작업 방지
- 요구사항의 분석, 명세, 검증 과정을 통해 소프트웨어 품질의 근간을 확보하고 개발 성공률 제고
- 요구사항의 변경을 효과적으로 관리하고 추적성을 유지하여 시스템의 유지보수성과 일관성 보장
 
## 구성요소
- 타당서 조사 : As-is 분석 효과분석, PoC
- 추출/도출(Elicitation): 이해관계자로부터 요구사항 수집 (인터뷰, 설문 등)
- 분석(Analysis): 요구사항 간 충돌 해결, 범위 확정, 우선순위 부여
- 명세(Specification): 정형/비정형 기법을 활용하여 문서화 (SRS 작성)
- 확인/검증(Validation/Verification): 요구사항이 올바른지, 사용자의 의도와 맞는지 검토
- 요구사항 관리(Management): 변경 제어, 추적성 관리(RTM), 형상 관리

## 구성도 요소
* 프로세스: 개발 프로세스(도출/분석/명세/확인) + 관리 프로세스
* 관계: SDLC 전 단계에 걸쳐 영향을 미치며, 특히 분석 단계의 핵심 활동임
```
[Elicitation] -> [Analysis] -> [Specification] -> [Validation]
       ^             |                |                |
       +-------------+------- [Management] ------------+
```

## 키워드
