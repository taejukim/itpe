#SW공학 #아키텍처 #패턴

## 정의
재사용 가능한 아키텍처 해법, 아키텍처 패턴
- 반복적으로 발생하는 아키텍처 설계 문제에 대한 검증된 해결책과 구조적 템플릿
- 시스템의 전체 구조를 조직화하고 하위시스템 간 관계를 정의하는 고수준 설계 패턴
## 키워드
- Layered, Client-Server, Microservices, Event-Driven
- Pipe-Filter, MVC, Broker, Peer-to-Peer
- 모듈화, 독립성, 재사용성, 확장성
## 암기법
- 구조 패턴 8가지: 레클마이 파브피
  (Layered-Client/Server-Microservices-Event-Driven-Pipe/Filter-Broker-P2P)
- 계층-분산-이벤트 (3대 범주)
## 목적
- 검증된 아키텍처 솔루션으로 개발 시간 단축
- 시스템 품질속성(확장성, 유지보수성) 향상
## 구성요소
- Layered Pattern(계층 패턴): 시스템을 계층으로 분리 (프레젠테이션-비즈니스-데이터)
- Client-Server Pattern: 서버가 서비스 제공, 클라이언트가 요청
- Microservices Pattern: 독립적으로 배포 가능한 작은 서비스들의 조합
- Event-Driven Pattern: 이벤트 생성/감지/소비로 통신
- Pipe-Filter Pattern: 데이터 스트림을 필터로 순차 처리
- MVC Pattern: Model-View-Controller 분리
- Broker Pattern: 중개자가 컴포넌트 간 통신 조정
- Peer-to-Peer Pattern: 동등한 노드들이 직접 통신
## 구성도
```
[Layered Pattern]
┌─────────────────┐
│ Presentation    │
├─────────────────┤
│ Business Logic  │
├─────────────────┤
│ Data Access     │
├─────────────────┤
│ Database        │
└─────────────────┘

[Microservices Pattern]
┌────────┐  ┌────────┐  ┌────────┐
│Service1│  │Service2│  │Service3│
└───┬────┘  └───┬────┘  └───┬────┘
    └───────────┼───────────┘
                ▼
         ┌────────────┐
         │ API Gateway│
         └────────────┘

[Event-Driven Pattern]
┌────────┐    Event    ┌────────┐
│Producer│─────────────►│Consumer│
└────────┘              └────────┘
              │
              ▼
         ┌─────────┐
         │Event Bus│
         └─────────┘

[Pipe-Filter Pattern]
Input → [Filter1] → [Filter2] → [Filter3] → Output
```
