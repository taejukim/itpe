#SW공학 #테스트 #테스트원칙 #품질관리

## 정의
소프트웨어 테스팅의 7가지 기본 원리
- ISTQB에서 제시하는 효과적인 테스트를 위한 보편적인 원칙
- 테스트 활동의 방향성과 한계를 이해하기 위한 기본 지침
## 키워드
- ISTQB, 7가지 원리, Defect Clustering
- Pesticide Paradox, Early Testing
- Exhaustive Testing, Context Dependent
- 결함 존재 입증, 완벽한 테스팅 불가능
## 암기법
- 7원칙: 결완조결살정오 (결함존재-완벽불가-조기-결함집중-살충제-정황의존-오류부재)
- 조기 테스팅 법칙: 1:10:100 (요구-설계-코딩 단계별 비용 증가)
- 결함 집중: 80:20 (파레토 법칙)
- 핵심: 테스트는 결함 발견이지 결함 부재 증명 아님
## 목적
- 효과적인 테스트 수행을 위한 기본 방향 제시
- 테스트의 한계와 가능성에 대한 올바른 이해 제공
## 구성요소
### 1. 테스팅은 결함이 존재함을 보여주는 활동 (Testing shows the presence of defects)
- 테스트는 결함을 발견할 수 있지만, 결함이 없음을 증명할 수는 없음
- 테스트는 결함의 수를 줄일 수 있지만 제로 결함을 보장하지 못함
### 2. 완벽한 테스팅은 불가능 (Exhaustive testing is impossible)
- 모든 조합을 테스트하는 것은 시간/비용상 불가능
- 위험도와 우선순위에 기반한 선택적 테스트 수행 필요
- 테스트 전략과 기법을 활용한 효율적 테스트 필요
### 3. 조기 테스팅 (Early testing)
- 테스트 활동은 소프트웨어 개발 생명주기 초기부터 시작
- 요구사항 분석 단계부터 테스트 계획 수립
- 초기 발견 시 수정 비용 절감 (1:10:100 법칙)
### 4. 결함 집중 (Defect clustering)
- 소수의 모듈에 대다수의 결함이 집중되는 경향
- 파레토 법칙(80:20): 20%의 모듈에 80%의 결함 발생
- 결함 밀집 영역에 테스트 노력 집중 필요
### 5. 살충제 패러독스 (Pesticide paradox)
- 동일한 테스트를 반복하면 새로운 결함 발견 불가
- 테스트 케이스를 주기적으로 검토하고 갱신 필요
- 새로운 테스트 기법과 시나리오 지속적 개발
### 6. 테스팅은 정황에 의존 (Testing is context dependent)
- 시스템의 특성, 도메인, 위험에 따라 테스트 접근법 변경
- 안전 필수 시스템과 일반 앱은 테스트 강도가 다름
- 프로젝트 상황에 맞는 적절한 테스트 전략 필요
### 7. 오류-부재의 궤변 (Absence-of-errors fallacy)
- 결함이 없다고 해서 좋은 소프트웨어는 아님
- 사용자 요구를 만족시키지 못하면 무의미
- 요구사항 검증(Validation)이 중요
