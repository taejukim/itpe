## 정의
수집된 요구사항을 명확하고 구체적으로 기록한 공식 문서, 요구사항 명세서
- 프로젝트 팀과 이해관계자 간의 합의된 요구사항을 기술한 문서로 개발 및 검정의 기준이 됨
- 기능 요구사항(무엇을 하는가)과 비기능 요구사항(어떻게 동작하는가)을 포함함

## 구성요소
- 기능 요구사항: 시스템이 수행해야 할 기능, 데이터 처리, 인터페이스 정의
- 비기능 요구사항: 성능, 가용성, 보안성, 유지보수성, 제약 사항 등 품질 특성
- 사용자 요구사항: 비즈니스 관점의 언어로 기술된 요구사항
- 시스템 요구사항: 기술적인 관점에서 상세화된 요구사항 (SRS: System Requirements Specification)
- 검증 기준: 각 요구사항이 충족되었는지 확인할 수 있는 Acceptance Criteria

## 구성도 요소
* 좋은 명세서의 특징: 완전성, 일관성, 명확성, 추적 가능성, 수정 가능성, 검증 가능성
* 관계: 요구사항 명세서 → 분석 및 설계 → 구현 → 테스트(V-모델 연계)
```
[User Request] -> [Requirements Specification] -> [Design] -> [Implementation]
                         |                                      ^
                         +----------- (Validation) -------------+
```

## 키워드
* SRS, 기능/비기능, 완전성, 추적성, 품질 속성

## 암기법
